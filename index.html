Introduction
A Model-Agnostic Pipeline to parseLab
face2parselab converts any structured message model into parseLab-compatible JSON — the input required by Lockheed Martin's parseLab secure parser generator. No MUDDL toolchain. No .NET runtime. No proprietary dependencies. ~350 lines of pure Python.
〈/〉architecture
# Two-layer design — readers and exporters
# are independently swappable

─── Layer 1: Model Readers ───────────────

FaceReader(skayl_path, templates_path)
  # Reads FACE UDDL .skayl models
  # Used by: SOSA, UCI, FACE-conformant systems

MAVLinkReader(xml_path)
  # Reads MAVLink common.xml definitions
  # Used by: UAS autopilots, GCS systems

# Future readers plug in here:
XSDReader   # XML Schema Definition
ProtoReader # Protocol Buffers


─── Common IR (format-agnostic) ──────────

class Protocol:
  structs:  list[Struct]   # nested types
  messages: list[Struct]   # top-level messages


─── Layer 2: Exporters ───────────────────

export_json(protocol) → parseLab JSON
# Future exporters plug in here:
# export_daedalus(), export_proto()...


─── CLI ──────────────────────────────────

python -m face2parselab run config.yaml
  # model in → parseLab JSON out
  # one command, any model format

Step 01 · Simple Model
Hand-Crafted Model — GPS Position Report
A minimal FACE UDDL model written by hand. 1 message, 1 nested struct, 7 fields — simple enough to read top-to-bottom and verify the output yourself. Demonstrates primitives, enumerations, range constraints, and variable-length strings.

  name="GpsCoordinatesType"
  xmi:type="platform:View">

  <characteristic rolename="Latitude"
    platformType="TYPE_DOUBLE"
    attributeKind="privatelyScoped"/>

  <characteristic rolename="Longitude"
    platformType="TYPE_DOUBLE"
    attributeKind="privatelyScoped"/>

  <characteristic rolename="Altitude"
    platformType="TYPE_DOUBLE"
    attributeKind="privatelyScoped"/>
</element>

<!-- Top-level message -->
<element
  name="PositionReportMDT"
  xmi:type="platform:View">

  <characteristic rolename="SystemId"
    platformType="TYPE_ULONG"/>

  <characteristic rolename="Timestamp"
    platformType="TYPE_ULONG"/>

  <characteristic rolename="Position"
    viewType="VIEW_COORDS"
    attributeKind="foreignReference"/>

    platformType="TYPE_STRING"
    optional="true"/>

</element>

Step 02 · Format Verification
parseLab's Required Input Format
parseLab is open source (Apache-2.0, github.com/lmco/parselab). Its protocol.json input format is publicly documented. Here we show the spec side-by-side with our output — and confirm they match.
〈/〉parseLab protocol.json spec (github.com/lmco/parselab)
// parseLab protocol.json — documented format
// Source: Lockheed Martin ATL, Apache-2.0
{
  // Optional: reusable nested type definitions
  "structs": [
    {
      "struct_name": "<type_name>",
      "members": [ <fields> ]
    }
  ],

  // Required: top-level message definitions
  "protocol_types": [
    {
      "name": "<message_name>",
      "fields": [ <fields> ]
    }
  ]
}

// Each field:
{
  "name":     "fieldname",    // required
  "type":     "U32",          // required
  "value":    "0|1|2|3",     // optional — enum
  "value":    "(0, 32)",      // optional — range
  "dependee": "true"          // optional — length field
}

// Valid primitive types:
// U8   U16  U32  U64
// I8   I16  I32  I64
// F32  D8
// U8[length_field]  ← variable-length array

// Nested struct reference:
{
  "name": "position",
  "type": "gpscoordinatestype"  // ← struct name
}

Step 03 · Second Model Format
MAVLink — A Different Model, Same Pipeline
MAVLink is the de facto UAS communication protocol, used by ArduPilot, PX4, and most autopilots. It defines messages in a public XML format at mavlink.io — completely different from FACE UDDL. Same pipeline. Same exporter. Same output format. New reader, no other changes.
〈/〉mavlink_subset.xml (public — mavlink.io)
<!-- MAVLink message definition format -->
<!-- Source: mavlink.io — public domain  -->

<mavlink>
  <enums>
    <enum name="MAV_TYPE">
      <entry name="MAV_TYPE_GENERIC"     value="0"/>
      <entry name="MAV_TYPE_FIXED_WING"  value="1"/>
      <entry name="MAV_TYPE_QUADROTOR"   value="2"/>
      <!-- ... 40 more types -->
    </enum>

    <enum name="MAV_STATE">
      <entry name="MAV_STATE_UNINIT"      value="0"/>
      <entry name="MAV_STATE_BOOT"        value="1"/>
      <entry name="MAV_STATE_CALIBRATING" value="2"/>
      <!-- ... -->
    </enum>
  </enums>

  <messages>
    <!-- HEARTBEAT — message ID 0 -->
    <message id="0" name="HEARTBEAT">
      <field type="uint8_t"  name="type"
             enum="MAV_TYPE">Vehicle type</field>
      <field type="uint8_t"  name="autopilot"
             enum="MAV_AUTOPILOT">Autopilot</field>
      <field type="uint8_t"  name="base_mode">Mode bitmap</field>
      <field type="uint32_t" name="custom_mode">Custom</field>
      <field type="uint8_t"  name="system_status"
             enum="MAV_STATE">Status</field>
      <field type="uint8_t"  name="mavlink_version">Version</field>
    </message>

    <!-- PARAM_SET — message ID 23 -->
    <message id="23" name="PARAM_SET">
      <field type="uint8_t"  name="target_system"/>
      <field type="uint8_t"  name="target_component"/>
      <field type="char[16]" name="param_id"/>
      <field type="float"    name="param_value"/>
      <field type="uint8_t"  name="param_type"
             enum="MAV_PARAM_TYPE"/>
    </message>

    <!-- COMMAND_LONG — message ID 76 -->
    <message id="76" name="COMMAND_LONG">
      <!-- target_system, target_component,
           command, confirmation,
           param1..param7 (float) -->
    </message>
  </messages>
</mavlink>
