Introduction
A Model-Agnostic Pipeline to parseLab
face2parselab converts any structured message model into parseLab-compatible JSON — replacing a toolchain that required .NET, Phenom, ANTLR, and proprietary DLLs with ~350 lines of pure Python. Steps 1–3 execute live in your browser. Step 4 fetches the MUDDL reference directly from GitHub and diffs it field by field — nothing is assumed.
〈/〉architecture
# Two-layer design — readers and exporters
# are independently swappable

─── Layer 1: Model Readers ───────────────
FaceReader(skayl_path, templates_path)
  # Reads FACE UDDL .skayl models
  # Used by: SOSA, UCI, FACE-conformant systems

MAVLinkReader(xml_path)
  # Reads MAVLink common.xml definitions

# Future: XSDReader, ProtoReader...


─── Common IR (format-agnostic) ──────────
class Protocol:
  structs:  list[Struct]
  messages: list[Struct]


─── Layer 2: Exporters ───────────────────
export_json(protocol) → parseLab JSON
# Future: export_daedalus(), export_proto()


─── CLI ──────────────────────────────────
python -m face2parselab run config.yaml

Step 02 · Format Proof ⚡ LIVE VERIFICATION
Does Our Output Actually Meet the parseLab Spec?
We take the JSON just generated in Step 1 and verify it live against parseLab's documented requirements — checking each requirement programmatically against the actual output, not a hardcoded table. parseLab is open source (Apache-2.0) at github.com/lmco/parselab.
〈/〉parseLab format requirements — github.com/lmco/parselab
// parseLab protocol.json — documented spec
// Lockheed Martin ATL · Apache-2.0

// Requirement 1: top-level keys
{
  "structs": [...],         // optional
  "protocol_types": [...]   // required
}

// Requirement 2: struct definition
{ "struct_name": "name", "members": [...] }

// Requirement 3: message definition
{ "name": "msg_name", "fields": [...] }

// Requirement 4: primitive field
{ "name": "fieldname", "type": "U32" }

// Requirement 5: valid primitive types
// U8 U16 U32 U64 I8 I16 I32 I64 F32 D8
// U8[length_field]  ← variable array

// Requirement 6: enumeration field
{ "name": "status", "type": "U32",
  "value": "0|1|2|3" }

// Requirement 7: range constraint field
{ "name": "count", "type": "U32",
  "value": "(0, 32)" }

// Requirement 8: variable-length array
{ "name": "datalength", "type": "U32",
  "dependee": "true" },
{ "name": "data",
  "type": "U8[datalength]" }

// Requirement 9: nested struct reference
{ "name": "position",
  "type": "gpscoordinatestype" }

Step 03 · MAVLink ⚡ LIVE EXECUTION
Different Model Format — Same Pipeline
MAVLink is the de facto UAS autopilot protocol — used by ArduPilot, PX4, and most ground control systems. Its XML format is completely different from FACE UDDL. The pipeline doesn't change. The exporter doesn't change. Only the reader changes — ~150 lines of Python. This is what adding ACC, CCA, or any new domain looks like.
〈/〉mavlink_subset.xml — fetched live from GitHub

Step 05 · Roadmap
What This Enables
Stage 2 is complete and verified. Stage 3 — wiring parseLab at the end to produce a compiled secure parser — is the next step. The architecture is already designed for it. Adding a new domain like ACC or CCA means one new reader, ~200 lines of Python.
〈/〉full chain — stage 3 target

  → SystemStatusMDT.json  ✓ live verified

  → Protocol → export_json()
  → heartbeat.json  ✓ live generated


# Stage 3 — NEXT: wire parseLab

  → Protocol
  → export_json(out="protocol.json")
  → subprocess.run([
      "python", "generate_parser.py",
      "--protocol", "protocol.json",
      "--module",   "hammer"
    ])
  → secure_parser.c  ← target
